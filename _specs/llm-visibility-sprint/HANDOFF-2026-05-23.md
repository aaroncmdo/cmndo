## Harte Regeln + Gotchas (diese Session teuer gelernt)
1. **Ich merge NIE selbst** → nur PR gegen `staging`, build-grün; die benannte **Merge-/Release-Session** (`kitta/merge-watcher` / `sync-staging-main-*`) zieht staging→main. Memory `feedback_staging_main_autonom`.
2. **CI-`build` hängt an Supabase** (`RLS-Function-Grants`-Step) → bei Prod-DB-Überlast (viele Parallel-Sessions/Release) **522-Fail = KEIN Code-Fehler**. Auch sporadischer `actions/checkout`-Auth-Flake. **Fix: re-run** (`gh run rerun <id> --failed`), sobald Supabase antwortet (`curl -s -o /dev/null -w "%{http_code}" https://paizkjajbuxxksdoycev.supabase.co/rest/v1/` → **401 = gesund**).
3. **Lokaler `next build` rot nur an `/gutachter-partner`** (Export-Timeout >60s) = Parallel-Session-DB-Contention, NICHT der eigene Code — wenn `✓ 30x/30x static pages` kam, ist alles gebaut. CI baut isoliert grün. `NODE_OPTIONS=--max-old-space-size=8192` Pflicht.
4. **307-Trap:** jede neue public Content-/Konversions-Route MUSS in middleware `publicPaths` + proxy `MARKETING_PREFIXES` (+ sitemap) — sonst 307→/login für anon+Crawler, **Build bleibt grün**. `next start` kaputt mit `output:standalone` → mit `next dev` smoken.
5. **Multi-PR-Stacking auf gleichen Routen** → add/add-Konflikt NUR im Import-Block (Render-Zeilen auto-mergen). Auflösung: alle Imports behalten, `git merge origin/staging` + Marker auflösen + push (kein Force). Mehrfach diese Session passiert (#1592).
6. **Eine FAQPage pro Seite:** FaqStems emittiert KEIN eigenes FAQPage-Schema mehr; `mergeFaqStemsIntoSchema` (jsonld) mergt in die vorhandene. Bei neuen FAQ-Quellen dieses Pattern nutzen.
7. **Worktree-Setup:** git-worktree hat kein `node_modules` (Junction auf Haupt-Repo) + kein `.env.local` (kopieren). Staging-Smoke: Basic-Auth-Creds `STAGING_BASIC_AUTH_USER/PASS` in `.env.local`; staging-Test gegen `https://app.staging.claimondo.de` (Memory `feedback_nur_staging_pullen`).
8. **Commit-Messages** via Bash-Heredoc + `git commit -F` (UTF-8, kein BOM).
