## 1 · CRITICAL — sofort fixen

### 1.1 🚨 `finance_eintraege` — Mass-Read+Write für jeden Authenticated

```
policyname: "Authenticated can manage finance_eintraege"
roles: authenticated · cmd: ALL · qual: true · with_check: true
```

Konsequenz: jeder eingeloggte User (Kunde, SV, Makler, externer Gast, jeder Account mit gültigem JWT) kann via PostgREST alle Finanz-Einträge lesen, einfügen, ändern, löschen. Das ist die Tabelle mit unseren Kosten/Einnahmen — Buchhaltungs-Daten.

**Fix:** Policy auf Staff/Admin/Dispatch beschränken (analog `aar_abrechnungen_rls_tighten`).

```sql
drop policy "Authenticated can manage finance_eintraege" on public.finance_eintraege;
create policy "Service-Role manage finance_eintraege" on public.finance_eintraege
  for all to service_role using (true) with check (true);
create policy "Staff read finance_eintraege" on public.finance_eintraege
  for select to authenticated
  using (exists (select 1 from profiles p where p.id=auth.uid() and p.rolle in ('admin','dispatch')));
```

Im 12.05.-Audit nicht aufgeführt — neuer Fund.

### 1.2 🚨 Public-Storage-Buckets mit sensiblen Inhalten — unverändert

| Bucket | `public` | Inhalt | Risiko |
|---|---|---|---|
| `fall-dokumente` | true | Vollmachten, Abtretungen, Polizeiberichte | Direkt-URL = Klartext-Read |
| `gutachten` | true | Gutachten-Anlagen | Direkt-URL = Klartext-Read |
| `schadensfotos` | true | Kennzeichen, Personen-Auto-Kontext | Direkt-URL = Klartext-Read |
| `unterschriften` | true | Handschrift-Unterschriften | Direkt-URL = Klartext-Read |

Diese Buckets sind nach 12.05.-Audit weiterhin offen. RLS-Policies auf `storage.objects` greifen nicht für `/storage/v1/object/public/...`-URLs — `public=true` ist das eigentliche Tor.

**Fix (pro Bucket):**
1. `update storage.buckets set public=false where id=...`
2. Code-Pfade auf `createSignedUrl` umstellen (Plan: `docs/13.05.2026/storage-rls-rollout-plan.md`).
3. Per-Bucket Read-Policy auf `(fall|sv|kanzlei|kunde)`-Scope statt `bucket_id = '...'`.

Zusätzlich: Bucket `unterschriften` hat Policy `"Anon can read own unterschriften"` mit `qual = (bucket_id = 'unterschriften')` — kein Filter auf Pfad/Eigentümer. Jeder anon kann ALLE Unterschriften listen, falls listing aktiviert wäre — durch `public=true` UND `name LIKE'flow/%'`-Pattern auch direkt erreichbar.

### 1.3 🚨 `mitteilungen_insert` + `phase_transitions_service_insert` — Spoofing-Pfade offen

```
mitteilungen_insert       — public/INSERT/with_check=true (kein Filter)
phase_transitions_service_insert — public/INSERT/with_check=true (kein Filter)
```

Konsequenz:
- `mitteilungen`: anon kann Mitteilungen mit beliebigem `empfaenger_id` einfügen → Impersonation in der Inbox eines fremden Users.
- `phase_transitions`: Audit-Trail-Spoofing — anon kann fake-Status-Wechsel mit beliebigem `fall_id` + Notizen erzeugen, die später Dispatch+Admin als „echte" Historie sehen.

**Fix:** INSERT-Policies auf `service_role` einschränken oder einen `with_check`-Constraint hinzufügen, der `auth.uid()`-Identität gegen die referenzierte Resource prüft.

```sql
drop policy "mitteilungen_insert" on public.mitteilungen;
create policy "mitteilungen_service_insert" on public.mitteilungen
  for insert to service_role with check (true);
-- Falls SVs/Kunden direkt schreiben sollen: separate Policy mit
-- with_check (sender_id = auth.uid() and exists(... fall_zugehoerig ...))

drop policy "phase_transitions_service_insert" on public.phase_transitions;
create policy "phase_transitions_service_only" on public.phase_transitions
  for insert to service_role with check (true);
```

## 3 · MEDIUM — Hygiene-Backlog

### 3.1 SECURITY DEFINER Functions im `public`-Schema (54 + 54 = 108 lints)

108 SECURITY-DEFINER-Functions sind für anon + authenticated EXECUTE-grantable. Klassischer Vektor wenn auch nur eine davon einen Bug hat, der Privilege-Escalation erlaubt. Cron-Jobs (z.B. `cron_dsgvo_hard_delete`, `cron_konsistenz_check`) sind besonders sensibel — die laufen mit postgres-Rechten und sollten von außen gar nicht aufrufbar sein.

**Fix-Strategie (analog Supabase-Best-Practice):**
1. SECURITY DEFINER Functions in eigenes Schema `private` verschieben (oder zumindest `internal`).
2. `revoke execute on function public.cron_* from anon, authenticated;`
3. Bei Functions die wirklich öffentlich brauchbar sein müssen (z.B. `is_admin()`, `can_access_fall()`): `SET search_path = pg_catalog, public` zwingend, plus minimaler erlaubter Aufruf-Scope.

### 3.2 `function_search_path_mutable` (64 lints)

64 Functions ohne `SET search_path = …` — search-path-Shadowing-Angriff möglich, wenn ein anderer User ein Objekt mit gleichem Namen in einem Schema mit höherer Priorität anlegen kann. Bei der Default-Permissions im `public`-Schema theoretisch ausnutzbar.

**Fix:**
```sql
alter function public.<name>(...) set search_path = pg_catalog, public;
```

Batch-Migration sinnvoll.

### 3.3 Public Bucket Allows Listing (6 lints)

Storage-Buckets mit `public=true` erlauben anon das Listen aller Objekte. Memory-Audit + AGENTS.md hat das schon als Backlog markiert, Fix-Plan in `docs/13.05.2026/storage-rls-rollout-plan.md`.

### 3.4 `auth_leaked_password_protection` deaktiviert (1 lint)

Supabase-Auth-Einstellung "Leaked Password Protection" ist aus → Nutzer können Passwörter setzen, die in HIBP-Datenbanken vorkommen.

**Fix:** Dashboard → Auth → Providers → Email → "Leaked Password Protection" auf "On" stellen. 1-Klick-Job.

### 3.5 `btree_gist` im public Schema (1 lint)

Extension `btree_gist` ist im `public`-Schema installiert. Best practice: in eigenes Schema (`extensions`) verschieben.

```sql
create schema if not exists extensions;
alter extension btree_gist set schema extensions;
```

(Mit Vorsicht testen — der `EXCLUSION CONSTRAINT` aus AAR-865 (gutachter_termine) hängt davon ab.)

## 4 · LOW / Performance

### 4.1 RLS-Performance — auth_rls_initplan (234 lints)

Sehr verbreitet: Policies wie `auth.uid() = user_id` werden für jede Row neu evaluiert, weil Postgres die Funktion nicht als stable+memoizable erkennt. Pattern-Fix:

```sql
-- Statt:
using (auth.uid() = user_id)
-- Schreiben:
using ((select auth.uid()) = user_id)
```

Bringt bei großen Tabellen 10–100× Speedup auf SELECT-Heavy-Queries. Ist eine Suchen-und-Ersetzen-Migration über alle Policies.

### 4.2 Multiple Permissive Policies (816 lints)

816× kommt es vor, dass mehrere Policies mit `permissive=true` auf derselben Tabelle/CMD/Role-Kombination existieren. Bei `SELECT` werden alle Permissive-Policies OR-verkettet → mehr Plan-Arbeit. Wo möglich: in eine Policy konsolidieren.

Hot-Spots (aus dem Coverage-Output, je 9 policies): `faelle`, `leads` — diese beiden Tabellen sind die häufigsten Reads im System. Konsolidierung lohnt.

### 4.3 Unindexed Foreign Keys (113 lints)

Wichtige Hot-Spots:

| Tabelle | FK | Auswirkung |
|---|---|---|
| `faelle` | `lead_id_fkey`, `konvertiert_von_lead_fkey`, `dispatch_id_fkey` | Join leads→faelle, Dispatch-View |
| `tasks` | `erstellt_von_id_fkey`, `gate_task_id_fkey` | Task-Subqueries |
| `nachrichten` | `sender_id_fkey`, `kb_empfaenger_id_fkey` | Inbox-Queries |
| `termine` + `gutachter_termine` | `fall_id_fkey` | Termine-Liste pro Fall |
| `zahlungspositionen` | `fall_id_fkey`, `zahlung_id_fkey` | Finance-Views |
| `whatsapp_inbound_messages` | `matched_lead_id_fkey`, `matched_termin_id_fkey` | Inbound-Routing |

Migrations-Vorschlag: ein `aar_perf_fk_indexes` Sweep, alle 113 in einer Migration.

### 4.4 Duplicate Indexes (2)

2 doppelte Indexes — Speicher- + Write-Overhead. Drop duplicates per Advisor-URL identifizieren und `drop index concurrently …`.

### 4.5 Unused Indexes (121 INFO lints)

Index gibt's aber wird nicht genutzt. Vor dem Drop: bestätigen dass die Stats-Daten nicht zu jung sind (Tabelle muss mindestens 1 Woche traffic gesehen haben) — sonst hatten neue Indexes noch keine Chance.
