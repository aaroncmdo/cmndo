# Security-Audit Claimondo-App — 12.05.2026

**Methodik:** 3 parallele Senior-Security-Subagenten, reine Lese-Analyse des Codes (kein DB-Zugriff, keine Exploit-Reproduktion). Scope: (1) Auth / Autorisierung / Token-Strecken / RLS-Migrations · (2) API-Routes / Webhooks / Crons / Server-Actions · (3) XSS / Injection / Secrets / Data-Exposure / externe Integrationen. Findings nur ab Confidence ≥ 8 in den Haupt-Report; alles darunter unter „Hardening".

> **⚠️ Wichtige Einschränkung:** Das **Basis-DB-Schema** liegt nicht im Repo — die frühen Supabase-Migrations sind leere `*_placeholder.sql` (Squash-Baseline). Die ursprünglichen `ENABLE ROW LEVEL SECURITY`-Statements + `GRANT`s für die Kerntabellen (`profiles`, `faelle`, `leads`, `nachrichten`, `flow_links`, `sachverstaendige`, `makler`, `makler_provisionen`, `provisionen_maik`, `abrechnungen`, `fall_dokumente`, …) sind aus dem Code **nicht verifizierbar**. Mehrere MEDIUM-Findings unten sind genau deshalb „RLS-abhängig": ob sie real ausnutzbar sind, hängt davon ab, ob diese Tabellen RLS-aktiviert sind und restriktive Policies haben. **Erste Empfehlung daher: ein Live-Schema-/RLS-Audit** (Supabase Security-Advisors bzw. `SELECT * FROM pg_policies` + Check welche Tabellen `rowsecurity=true` haben + welche `GRANT … TO anon/authenticated` existieren). Ohne das bleibt die RLS-Schicht ein blinder Fleck.

## HIGH

### H1 — Twilio-Inbound-Webhook ohne Signatur-Verifizierung
**Datei:** `src/app/api/webhooks/twilio/inbound/route.ts` (~Z. 74). Auch betroffen, geringere Auswirkung: `src/app/api/webhooks/twilio/status/route.ts`.
**Kategorie:** Missing webhook authentication → unauthenticated state mutation + Stored-XSS-Vektor + SSRF.
**Befund:** Der WhatsApp-Inbound-Webhook parst `req.formData()` und handelt darauf **ohne jede `X-Twilio-Signature`-Prüfung** (die `TWILIO_AUTH_TOKEN`-Referenzen in der Datei dienen nur dem Media-Download, nicht der Request-Auth). Er ist der schreib-intensivste Webhook der App: nutzt den **Supabase-Admin-Client (RLS-Bypass)** für Inserts/Updates auf `whatsapp_inbound_messages`, `gutachter_termine.status` (→ `'bestaetigt'`), `timeline`, `fall_dokumente`, `leads.*` (zb1_status, polizeibericht_status, bevorzugter_kanal, OCR-extrahierte Fahrzeugdaten), `benachrichtigungen`, und triggert **ausgehende WhatsApp-Sends** an die `From`-Nummer. Bei `NumMedia≥1` lädt er die `MediaUrl{n}` herunter (Pfad/Host vom Request bestimmt → SSRF-Vektor). Die Schwester-Route `src/app/api/twilio/inbound-kb-whatsapp/route.ts` **validiert** die Signatur korrekt — hier wurde es schlicht vergessen.
**Exploit:** Angreifer kennt die Webhook-URL → POST `From=whatsapp:+49<Opfer>&MessageSid=<random>&Body=JA` → `matchInboundToFall` ordnet die Nummer einem offenen Lead/Fall mit kommendem Gutachter-Termin zu → Termin wird force-bestätigt + Bestätigungs-WhatsApp an die gespoofte Nummer + Timeline-Eintrag mit Angreifer-Text in der Fallakte + KB-Benachrichtigung mit Angreifer-Text. Variieren von `Body`/`NumMedia`/`MediaContentType{n}` → Timeline-Pollution, `bevorzugter_kanal`-Manipulation, Fake-Dokument an fremdem Lead, beliebige Outbound-WhatsApp vom Firmen-Account, Server-Side-Fetch beliebiger URLs.
**Fix:** Vor jeder Verarbeitung `X-Twilio-Signature` gegen `TWILIO_AUTH_TOKEN` + vollständige Request-URL + sortierte Form-Params validieren (`twilio.validateRequest(...)` bzw. die vorhandene `validateTwilioSignature`-Logik aus `inbound-kb-whatsapp/route.ts` wiederverwenden); bei ungültig/fehlend → 403. Dasselbe für `webhooks/twilio/status/route.ts`. **(Confidence 9 — von zwei unabhängigen Audit-Pässen identifiziert.)**

## MEDIUM

### M1 — IDOR: Pre-Call-Briefing leakt Fall-Daten an jeden eingeloggten User
**Datei:** `src/app/api/copilot/briefing/route.ts` (~Z. 11), `src/lib/copilot/briefing.ts` (`loadBriefingContext`, ~Z. 42).
**Kategorie:** Broken access control / IDOR / PII-Leak.
**Befund:** `POST /api/copilot/briefing` prüft nur `supabase.auth.getUser()` (= *irgendein* eingeloggter User), nimmt dann den vom Client gelieferten `fallId`/`leadId` und gibt ihn an `loadBriefingContext()` weiter — das per **`createAdminClient()` (RLS-Bypass)** liest: `v_faelle_mit_aktuellem_termin`, den verknüpften `leads`-Datensatz (Kundenname), `gutachten_betrag` (Schadenshöhe), Fall-Status, Stepper-Phase, die **letzten 3 `nachrichten` des Falls**, und die neueste `fall_summaries`-KI-Zusammenfassung — und streamt daraus eine LLM-Briefing-Antwort zurück. **Kein Ownership-/Rollen-Check auf `fallId`.** Jeder Account (Kunde, SV, Makler, Kanzlei) kann Briefing-Daten zu jedem Fall lesen.
**Exploit:** Eingeloggter Makler/SV (kennt legitim einige `fallId`s, kann plausibel weitere erlangen) POSTet `{"fallId":"<fremde-UUID>"}` → bekommt Kundenname, Fahrzeug, Schadenshöhe, letzte Chat-Nachrichten und die KI-Zusammenfassung eines Falls ohne jede Beziehung. (Begrenzt durch `fallId`=UUID, aber das Fehlen der Autz ist real.)
**Fix:** Vor dem Laden prüfen, dass der Caller den Fall sehen darf — z. B. `requireRole(['admin','dispatch','kundenbetreuer'])` (das Briefing ist ein KB/Dispatch-Pre-Call-Tool), oder `kundenbetreuer_id`/`sv_id`/`makler_id` gegen den User checken, oder den Read mit dem User-Context-Client machen (RLS greift). **(Confidence 8.)**

### M2 — Finanz-Server-Actions ohne Rollen-Guard (Maik-Provisionen + SV-Werbebudget)
**Datei:** `src/app/admin/finance/(hub)/provisionen/actions.ts` (`setCpl`, `confirmProvision`, `markMonthAsPaid`, `reverseProvision` — alle ~ab Z. 7). Verwandt: `src/app/admin/finance/actions.ts` (`erfasseEinzahlung`, ~Z. 6).
**Kategorie:** Missing authorization / privilege escalation (Finanz-Integrität).
**Befund:** Diese Actions prüfen nur `if (!user)`, **nicht** ob der Caller Admin ist. Sie liegen unter `/admin/finance/...`, aber `admin/layout.tsx` schützt **keine Server-Actions** — die sind eigenständig per POST aufrufbar (Action-IDs ableitbar). `confirmProvision` flippt `provisionen_maik` `pending→confirmed` (der `maik-monatsabrechnung`-Cron zahlt dann alle `confirmed`-Zeilen eines Monats aus); `markMonthAsPaid` flippt bulk `confirmed→paid`; `reverseProvision` voided Einträge; `setCpl` setzt beliebige Cost-per-Lead. `erfasseEinzahlung` erhöht `sachverstaendige.werbebudget_guthaben_netto` für einen beliebigen `sv_id`. **Alle anderen Admin-Actions im Repo** (`admin/tasks`, `admin/vertraege`, `admin/abrechnungen`, `admin/sachverstaendige/[id]`, `lib/actions/dsgvo-loeschung`, …) machen `requireRole(['admin'])` — diese Files sind die Ausreißer, entgegen dem in AGENTS.md vorgeschriebenen Pattern.
**Exploit:** Beliebiger eingeloggter User (z. B. `kunde`) ruft `confirmProvision('<id>')` / `markMonthAsPaid('2026-04')` → schiebt Partner-Provisionen in die Auszahlungs-Pipeline; oder `reverseProvision('<id>','...')` → voided legitime Provisionen. Real ausnutzbar wenn `provisionen_maik` / `sachverstaendige` keine restriktive RLS-UPDATE-Policy haben (Schreibzugriff läuft über den User-Context-Client).
**Fix:** `const guard = await requireRole(['admin']); if (!guard.success) return { success:false, error: guard.error }` an den Anfang jeder Action in beiden Files; zusätzlich RLS auf `provisionen_maik` + `sachverstaendige` als Defense-in-Depth verifizieren. **(Confidence 8 für das Fehlen des Guards; reale Auswirkung RLS-abhängig.)**

## Geprüft & als sicher bewertet (keine Action nötig)

- **Cron-Routes:** alle `src/app/api/cron/**/route.ts` prüfen `Authorization: Bearer ${CRON_SECRET}` (per grep verifiziert — keine fehlt).
- **Webhook-Signaturen (korrekt):** `stripe/webhook` (`stripe-signature`), `webhooks/lexdrive` + `lexdrive/bot-callback` + `lexdrive/vollmacht-confirm` (HMAC-SHA256, `timingSafeEqual`), `baileys/inbound` (Bearer), `twilio/inbound-kb-whatsapp` (`x-twilio-signature` in Prod), `aircall/webhook` + `webhooks/aircall/inbound` (HMAC wenn Secret gesetzt).
- **Auth-Layout-Guards:** `admin`, `dispatch`, `gutachter`, `kunde`, `makler/(shell)`, `mitarbeiter`, `kanzlei`, `faelle/[id]` — alle prüfen `getUser()` + Rolle; Query-Errors rutschen nicht durch (`.single()`/`.maybeSingle()` → `data:null` → der `profile?.rolle !== '<x>'`-Check greift → Redirect); `admin/layout.tsx` prüft `profileErr` explizit. `portal-guard.ts` behandelt den `error`-Pfad korrekt.
- **Middleware:** 2FA-Check vor Admin-Rollen-Check (AAR-111 intakt); Dispatch-User von `/admin/*` weggeleitet; `getUser()` (HTTP-Roundtrip) statt `getSession()` für geschützte Pfade.
- **Service-Role-Key:** `createAdminClient()`/`createServiceClient()` ausschließlich in Server-Components / `'use server'`-Files / API-Routes — kein Import in `'use client'`-Dateien, kein Leak in API-Responses.
- **`NEXT_PUBLIC_*`:** keine Variable enthält ein Secret — nur legitime Public-Tokens (Mapbox `pk.`, Google-Maps-Browser-Key, Supabase-URL+Anon-Key, Cesium-Ion-Token, Imagin-Customer-ID, App-URLs, Feature-Flags). `VERCEL_API_TOKEN`/`*_SECRET`/`ANTHROPIC_API_KEY`/`TWILIO_AUTH_TOKEN`/`STRIPE_*_SECRET`/`*_WEBHOOK_SECRET`/`CRON_SECRET` bleiben server-only.
- **Kein Secret-/Passwort-/Token-Logging** — `console.error/warn` in Token-Nähe loggen nur Error-Objekte/Messages/Status-Labels.
- **2FA:** Email-OTP SHA-256-gehasht, 5-Min-Expiry, Single-Use, 3/h-Rate-Limit; SMS-2FA via Twilio Verify (keine Codes in DB); `skip-cookie.ts` mit Defensive-Re-Check.
- **Remember-Me-Token:** `randomBytes(32).base64url`, SHA-256-gehasht in DB, 30-Tage-Expiry, `revoked_am`-Check, Cookie `httpOnly`+`secure`, User-ID-Binding gegen Hash geprüft.
- **Token-Generierung (im App-Code sichtbar):** durchgehend `crypto.randomBytes(24..32).base64url`/`.hex` (192–256 bit) — kein `Math.random`. Token-Routen prüfen Expiry, teils Single-Use.
- **Passwort-Reset:** komplett über Supabase-GoTrue-Recovery-Token (kein eigenes Token-Handling), `signOut()` danach.
- **OAuth-Callbacks:** `auth/callback` ignoriert `next` (kein Open-Redirect); `auth/google/callback` + `auth/google-calendar/callback` whitelisten `returnTo`/`state` auf relative Pfade. `auth/logout` 303 mit X-Forwarded-Origin.
- **`dangerouslySetInnerHTML`-Treffer (außer L7):** JSON-LD (`jsonLdScript()`, serialisiertes eigenes Objekt), QR-SVGs aus der `qrcode`-Library (deterministisch, kodiert URLs), hardcoded `SVG_INLINE`-Konstante, `vertragsvorlagen.inhalt_html` (nur Admin per Action mit Rollen-Guard schreibbar), `onboarding_felder.label` (nur per Migration geseedet).
- **Markdown-Rendering:** `react-markdown` in `LegalDocBody.tsx`/`MaklerCopilotTab.tsx` — kein `rehype-raw`/`skipHtml=false`, HTML wird escaped.
- **Kein `eval`/`new Function`/`child_process`/`exec` mit User-Input im Runtime-Code** (`.exec()`-Treffer sind alle `RegExp.prototype.exec`).
- **Supabase-`.rpc()`** (außer Seed-Script): durchweg gebundene Parameter — keine SQL-Injection.
- **Branding-Custom-Theme:** `primary/secondary/accent` als Hex validiert; alle Theme-Tokens fließen in `generateCssVars()` → React-`CSSProperties`-Objekt, nicht in rohes `<style>` — keine CSS-/Selector-Injection.
- **Webhook-Payload-Text** (z. B. LexDrive `begruendung`) landet in `timeline.beschreibung`, wird aber nur als JSX-Text (auto-escaped) gerendert — kein Stored-XSS.
- **Profile-Update-Actions** (`kunde/profil`, `gutachter/profil`, `lib/actions/sv/update-own-profile`): explizite `.eq('id', user.id)`, explizite Feld-Listen — kein Mass-Assignment, `rolle`/Privilege-Felder nicht schreibbar.
- **`api/dev/*` + `seed-testdata`:** durch `NODE_ENV==='development'` bzw. Admin-Rolle gegated.
- **`sv/upload-gutachten` + `.../finalize`:** verifiziert SV-Besitz von `auftragId`; `finalize` erzwingt `claim/<claimId>/gutachten/<auftragId>/`-Storage-Path-Whitelist.

## Priorisierte Maßnahmen

1. **Sofort (HIGH):** `X-Twilio-Signature`-Verifizierung in `webhooks/twilio/inbound` + `webhooks/twilio/status` nachrüsten (Template `validateTwilioSignature` aus `inbound-kb-whatsapp/route.ts` existiert). → kleiner, contained Fix.
2. **Bald (MEDIUM):** `requireRole`-Guards in `admin/finance/(hub)/provisionen/actions.ts` + `admin/finance/actions.ts`; Autz-Check in `copilot/briefing` (Rolle oder User-Context-Client).
3. **RLS-Verifikation (Blinder Fleck schließen):** Live-Schema-Dump — welche Tabellen `rowsecurity=true`, welche Policies, welche `GRANT … TO anon/authenticated`. Davon hängen M2 + L5 + mehrere L-Punkte ab. Danach ggf. eigenes RLS-Audit-Ticket.
4. **Hardening-Backlog:** L1 (Token-Hashing vereinheitlichen), L7 (SVG-Sanitization), L5/L6 (Rollen-Guard + Filter-Escaping auf `/api/search`), L13 (Webhook-Fail-Closed bei fehlendem Secret), L14 (`exec_sql`-Grant prüfen/entfernen), L9–L12 (Auth/Ownership-Checks auf OCR-/Email-/Aircall-/Lead-Routes).

*Hinweis: Dieser Report dokumentiert teils ausnutzbare Schwächen — bewusst mit Befund + Exploit + Fix. Privates Repo; Report dient als Tracker. Nach Behebung der HIGH-/MEDIUM-Punkte kann er als historischer Stand committet bleiben.*
