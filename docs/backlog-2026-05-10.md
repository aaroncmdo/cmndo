### Unfall-Vor-Ort-Aufnahme — Stufe 2 (RN-App + NFC + Verursacher) ⭐ Aaron-Briefing 10.05.

**Vision:** Geschädigter und Verursacher tauschen sich am Unfallort in 30 Sekunden statt 30 Minuten + Streit aus. NFC-Tap statt Zettel, Foto, Name, KFZ-Schein abfotografieren. Wir bauen die Brücke zwischen beiden — und kommen damit als erste in jedem Schadensfall an.

**Geschädigter-Flow** (App + Web):
1. „Unfall jetzt aufnehmen"-Button im Hero / im Kunde-Portal
2. **Kurze Vor-Ort-Beratung** — Live-Chat oder KI-Voice-Briefing („Was tun in den ersten 15 Minuten")
3. **Foto-Aufnahme-Wizard** — geleitete Sequenz (Übersicht / Schaden nah / Kennzeichen / Umfeld), GPS-Stempel, Zeit-Stempel, EXIF beibehalten
4. **NFC-Tap an Gegner-Phone** — App schreibt eindeutigen Link (`/u/{token}`) auf NFC-Tag des eigenen Phones, Gegner tappt rein
5. (Optional) Schaden direkt an Gegner-Versicherung melden — wenn deren Daten vorliegen

**Verursacher-Flow** (Web, ohne App):
1. NFC-Tap öffnet `/u/{token}` im Browser des Verursachers
2. Schritt 1: Telefonnummer + Name eingeben
3. Schritt 2: „Ich habe den Schaden verursacht" bestätigen (mit Hinweis auf Konsequenzen)
4. Schritt 3: Versicherungsdaten eingeben (Versicherer + Vers.-Nummer + ggf. Foto Versicherungs-Karte)
5. Schritt 4: Verursacher bekommt **Read-Only-Link** für Fall-Verlauf
6. Magic-Link an Email/SMS für Wiederbesuch

**Backend-Architektur:**
- Neue Tabelle `unfall_aufnahmen` mit FK zu Lead/Fall
- Felder: `gps_lat/lng`, `aufgenommen_am`, `fotos_jsonb` (URLs), `gegner_token`, `gegner_status`
- Neue Tabelle `unfall_gegner_eingaben` — Verursacher-Daten getrennt von eigentlichem Lead damit Datenschutz sauber bleibt
- Token-System: `gegner_token` ist UUID, public-readable, Read-Only auf Status-Subset
- Direktmeldung an Versicherer: Salesforce/B2B-API der jeweiligen Versicherung (HUK, AXA, LVM haben APIs — separates Spike)

**Komplexitäts-Stufen:**
- **MVP** (1-2 Wochen): Foto-Aufnahme + NFC-Web-Flow ohne native App — Web Share / NFC API auf Android funktioniert via Browser
- **Stufe 2** (2-4 Wochen): Native NFC-Schreiben in iOS App + Read-Only-Portal für Verursacher
- **Stufe 3** (4-6 Wochen): Direktmeldung an Versicherer-APIs + Vor-Ort-Beratung-KI + Real-Time-Chat zwischen Geschädigtem und Verursacher

**Datenschutz-Sensibel:**
- Verursacher gibt freiwillig Daten an — explizite Einwilligung
- Read-Only-Link für ihn rechtlich sauber (Art. 6 DSGVO „berechtigtes Interesse")
- Schuld-Bekenntnis muss als Beweismittel valide sein → ggf. Zeitstempel + GPS + ggf. Audio-Aufnahme „Ich habe den Schaden verursacht"

**Hebel:** Wenn das funktioniert, sind wir der **erste Anbieter überhaupt** der diesen Cross-Party-Flow hat. Wettbewerber haben nur Geschädigten-Seite. Marketing-Story: „Vom Unfall zum Anwalt in 60 Sekunden — beide Seiten."

### Baileys statt Twilio + Meta für WhatsApp ⭐ Aaron-Briefing 10.05.

**Vision:** Baileys (open-source WhatsApp-Web-Library) läuft als persistent Node-Service auf dem VPS. Damit:
- Keine Per-Message-Kosten (Twilio kostet ~5-10 Cent/Nachricht in DE)
- Kein Meta-Approval-Prozess (zieht sich Wochen)
- Alle Features — Status, Read-Receipts, Voice-Notes, Typing-Indicators
- Eigene Phone-Number-Validation („Ist diese Nummer WhatsApp-fähig?") via Baileys' Profile-Lookup

**Architektur:**

```
┌─ Next.js App (Vercel-CI / VPS) ──────────────┐
│                                              │
│  src/lib/whatsapp/                           │
│   ├── baileys-client.ts  (HTTP-Client zum    │
│   │                       Baileys-Service)   │
│   └── send.ts            (Send-Wrapper)      │
└──────────────┬───────────────────────────────┘
               │ HTTP / Internal Network
               ▼
┌─ Baileys-Service (auf VPS, eigener PM2) ────┐
│  - Persistent WhatsApp-Web-Connection        │
│  - Auth-Session in /var/lib/baileys/         │
│  - Express-Endpoints:                        │
│    POST /send       (sendMessage)            │
│    GET  /check/:nr  (isOnWhatsApp)           │
│    GET  /status     (connection-state)       │
│    POST /webhook    (incoming-Hook → next.js)│
└──────────────────────────────────────────────┘
```

**Risiken (klar dokumentiert):**

1. **Banning-Risiko** — Baileys nutzt WhatsApp-Web-Protocol, das ist gegen WA-ToS. Bei Mass-Sending oder Spam-Reports → permanenter Ban der Nummer
   - Mitigation: dedizierte Business-Nummer (separater SIM), Rate-Limiting, Opt-In-Logik
2. **Re-Auth bei Disconnect** — alle ~14 Tage muss QR-Code neu gescannt werden
   - Mitigation: Admin-Notification bei Disconnect, Slack-Alert
3. **Skaliert nur bedingt** — 1 Baileys-Service = 1 WhatsApp-Account. Bei mehreren Filialen / Brand-Lines brauchen wir mehrere Services
4. **Inoffiziell** — kein WhatsApp-Support, bei API-Changes sind wir auf den Open-Source-Maintainer angewiesen

**Migration-Plan (3 Phasen):**

**Phase 1 — Read-Only-Spike (1 Tag):**
- Baileys-Service auf VPS installieren als PM2
- QR-Code-Scan via Aaron's Phone (eigene Nummer für Test)
- `isOnWhatsApp(+49xxx)`-Endpoint live → Next.js kann Telefon-Validation machen
- Ersetzt Twilio-Lookup-API für Nummer-Check (spart Geld)

**Phase 2 — Outbound parallel zu Twilio (2-3 Tage):**
- `POST /send` Endpoint mit Templates
- In `src/lib/communications/send-fall.ts` Feature-Flag `WHATSAPP_PROVIDER=baileys|twilio`
- Pro Nachricht entscheiden: kritisch (legal/SA) → Twilio (Compliance), normal → Baileys
- Logging beider Provider parallel um Delivery-Rate zu vergleichen

**Phase 3 — Twilio + Meta abschalten (1 Woche):**
- Bei stabiler Baileys-Performance: Twilio-Templates ausphasen
- Meta-Cloud-API komplett raus
- Nur noch Aircall (Voice) + Resend (Email) + Baileys (WhatsApp) als externe Channels

**Aufwand:** ~1 Woche bis Phase 2 stabil, weitere 1-2 Wochen für volle Migration.

**Hebel:**
- Cost: 5-10 Cent × ~1000 Nachrichten/Monat = ~50-100 €/Monat Ersparnis (skaliert mit Volumen)
- Speed: Twilio-Approval entfällt für neue Templates
- Features: Read-Receipts geben uns echte Open-Rates (Twilio liefert nur Delivery-Status)

### Inbox-Refactor: Gruppen-Chat + 1:1-WhatsApp-Modus ⭐ Aaron-Briefing 10.05.

Aktuelle Multi-Channel-Inbox (Memory: `project_multi_channel_inbox.md`) hat 5 Kanal-Tabs pro Fall — alle parallel (Email/WhatsApp/SMS/Chat/Anruf). Das ist nice für Übersicht, aber unübersichtlich beim aktiven Schreiben.

**Neu — zwei Modi parallel:**

**Modus A: Gruppen-Chat pro Fall (Default-Inbox-View)**
- Ein Thread, alle Beteiligten drin: Kunde + KB + SV + Anwalt + ggf. Werkstatt
- Jeder Beitrag sieht den Channel woraus er kam (Icon: WA / Email / In-App)
- Vorteil: keiner muss switchen, ein Scroll = ganzer Verlauf
- Standard für „normale" Fall-Kommunikation

**Modus B: 1:1-WhatsApp direkt aus der App**
- Per Klick auf Beteiligten → eigener 1:1-Thread im WhatsApp-Style
- Nachrichten gehen out via Baileys (siehe Baileys-Block oben), kommen in als Inbound-Webhook
- Wir adressieren beliebige Telefonnummern — nicht nur Kunden, auch Werkstatt, Versicherer, Polizei
- Vorteil: privat, gezielt, ohne dass alle anderen mitlesen

**UI-Skizze (Fallakte → Inbox-Tab):**

```
┌── Inbox · Fall #ABC-123 ────────────────────┐
│                                              │
│  [Gruppe]  [Kunde]  [SV]  [Anwalt]  [+Neu]  │
│  ─────                                       │
│                                              │
│  Default = Gruppen-Thread aller Beteiligten  │
│  Klick auf Person-Tab = 1:1-WA-Thread        │
│  „+Neu" = WA-Thread zu beliebiger Nummer     │
│                                              │
└──────────────────────────────────────────────┘
```

**DB-Erweiterung:**
- `nachrichten` Table existiert (Memory) → neues Feld `thread_typ` enum `gruppe|direkt`
- Bei `direkt`: `partner_telefon` + `partner_rolle_label`
- Index auf (`fall_id`, `thread_typ`, `partner_telefon`) für schnellen 1:1-Tab-Wechsel

**Abhängigkeit:**
- 1:1-WhatsApp braucht **Baileys live** (siehe Block oben, Phase 2)
- Mit Twilio möglich, aber teurer pro Nachricht und braucht Template-Approval pro neuem Empfänger-Typ

**Aufwand:** 3-4 Tage (UI-Refactor + DB + Baileys-Integration)

**Hebel:** Mitarbeiter-Productivity. Heute switchen KB/SV ständig zwischen WA-Web, Email-Client und unserer Plattform. Mit beidem in einer Inbox: alles drin, mit Audit-Trail.
