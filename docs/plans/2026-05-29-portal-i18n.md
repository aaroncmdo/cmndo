# Portal-i18n (Kunde + Magic-Link, nutzerbasiert) — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (empfohlen) oder superpowers:executing-plans, um diesen Plan Task-für-Task umzusetzen. Steps nutzen Checkbox-Syntax (`- [ ]`).

**Goal:** Das Kunde-Portal und die Magic-Link-Strecken (`/flow`, `/upload/*`) nutzerbasiert in alle 6 Locales übersetzen — Sprache pro Nutzer in `profiles.sprache`, still aus dem Lead vorbelegt, im Portal wechselbar — plus On-Demand-Maschinenübersetzung für Falldaten als gekennzeichnete Anzeige-Hilfe.

**Architecture:** Eine einzige source-aware Auflösung in `src/i18n/request.ts` wählt je Route die Locale-Quelle (Cookie für Marketing, `profiles.sprache` fürs auth. Kunde-Portal, Token→`leads.sprache` für Magic-Link). Das spiegelt das bestehende Branding-Resolver-Muster und vermeidet die verifizierte Falle, dass ein nested `NextIntlClientProvider` nur Client-Komponenten re-lokalisiert. Übersetzungen kommen aus der bestehenden Anthropic-Claude-Pipeline; Falldaten-MT läuft über einen content-adressierten Cache (`content_translations`) mit server-seitigem Zugriff.

**Tech Stack:** Next.js 16.2.1 App Router · next-intl v4.9.1 (Cookie + profiles.sprache, kein URL-Prefix) · Supabase (CLI-Migrationen) · Anthropic `claude-sonnet-4-6` · Vitest + Playwright.

**Vollständige Spec:** `_specs/portal-i18n/` (CONTEXT · CONTRACT · DB_MIGRATION · WELLEN_PLAN). Dieser Plan ist der narrative Fahrplan + ausbuchstabierte Welle 1; die Wellen-Execution-Prompts stehen in `WELLEN_PLAN.md`.

```ts
export type LocaleSource = 'cookie' | 'profile' | 'token'
const TOKEN_PREFIXES = [
  ['flow', '/flow/'], ['upload-dokumente', '/upload/dokumente/'],
  ['upload-zb1', '/upload/zb1/'], ['re-termin', '/kunde/re-termin/'],
  ['ablehnen', '/ablehnen/'],
] as const
const TOKEN_EXACT = ['/kunde-termin', '/kunde/termin', '/sv'] // ohne Token-Segment, aber token-source
export function extractTokenFromPath(pathname: string | null) {
  if (!pathname) return null
  for (const [kind, prefix] of TOKEN_PREFIXES) {
    if (pathname.startsWith(prefix)) {
      const token = pathname.slice(prefix.length).split('/')[0]
      if (token) return { kind, token }
    }
  }
  return null
}
export function classifyLocaleSource(pathname: string | null): LocaleSource {
  if (!pathname) return 'cookie'
  if (extractTokenFromPath(pathname)) return 'token'
  if (TOKEN_EXACT.some((p) => pathname === p || pathname.startsWith(p + '/'))) return 'token'
  if (pathname === '/kunde' || pathname.startsWith('/kunde/')) return 'profile'
  return 'cookie'
}
```

## Risiken & Gegenmaßnahmen

| Risiko | Gegenmaßnahme |
|---|---|
| `request.ts` verlangsamt Marketing (DB-Read auf kritischem Pfad) | `classifyLocaleSource` liefert für Marketing/login `'cookie'` → kein DB-Read; `cache()`; Verify per Log. (B3) |
| `leads.sprache`-Format unbekannt → Fehl-Mapping | Welle-0-Live-Check + `normalizeToLocale` mit Alias-Map; unbekannt → `null`-Fallback, nie raten. |
| Server-Komponente falsch lokalisiert trotz Provider | Auflösung ausschließlich in `request.ts`; **kein** `setRequestLocale` (Race-Risiko, AAR-600). (E1) |
| `content_translations` Cache-Poisoning / Leak | RLS an, keine Client-Policies; Zugriff nur service-role im Server-Action; content-adressiert (Hash = Capability). (B6) |
| MT-Text wird rechtlich verwendet | Original = SSoT, MT klar gekennzeichnet, nie in PDF/SA/Email (B1; grep-Gate in W5). |
| DDL-Drift durch parallele Sessions | `information_schema` live vor Migration; nur CLI (Regel 2); Stash-Hygiene (Regel 3). |
| URL-Prefix versehentlich aktiviert → Routing kaputt | `next.config.ts`/`proxy.ts` sind Tabu (CONTEXT §2). |
